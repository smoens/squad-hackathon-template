# Ceremonies

> Hackathon-specific ceremonies — tuned for high-velocity delivery over 1–3 days.

## Morning Alignment

| Field | Value |
|-------|-------|
| **Trigger** | scheduled |
| **When** | Daily, before work begins |
| **Condition** | Every hackathon day |
| **Facilitator** | Jared |
| **Participants** | all |
| **Time budget** | 15 min max |
| **Enabled** | ✅ yes |

**Agenda:**
1. Yesterday's progress (or pre-hackathon status on Day 1)
2. Today's goals and stage focus
3. Blockers — especially data and access
4. Time-box reminders for the day

## Midday Checkpoint

| Field | Value |
|-------|-------|
| **Trigger** | scheduled |
| **When** | Daily, midpoint of work session |
| **Condition** | Every hackathon day |
| **Facilitator** | Jared |
| **Participants** | all |
| **Time budget** | 15 min max |
| **Enabled** | ✅ yes |

**Agenda:**
1. Progress check against morning goals
2. **Go / Pivot / Kill** decision on each active use case
3. Resource reallocation if needed
4. Escalation check — anything for Sarah?
5. Erlich: demo narrative status (is the story forming?)

## Demo Prep

| Field | Value |
|-------|-------|
| **Trigger** | scheduled |
| **When** | Before final demo session |
| **Condition** | Once per hackathon, during Demo stage |
| **Facilitator** | Erlich |
| **Participants** | all |
| **Time budget** | 30 min |
| **Enabled** | ✅ yes |

**Agenda:**
1. Demo flow walkthrough — what we show, in what order
2. Narrative arc — problem → insight → solution → impact
3. Dry-run with Richard (ensure PoC stability)
4. Backup plan if live demo fails
5. Q&A preparation — anticipated customer questions

## Post-Hackathon Retro

| Field | Value |
|-------|-------|
| **Trigger** | scheduled |
| **When** | After wind-down stage completes |
| **Condition** | Once per hackathon |
| **Facilitator** | Jared |
| **Participants** | all |
| **Time budget** | 30 min |
| **Enabled** | ✅ yes |

**Agenda:**
1. What went well?
2. What didn't work?
3. What should we do differently next time?
4. Skill extraction: reusable patterns for `.squad/skills/`
5. Knowledge capture for HQ sync (Scribe compiles)
6. Customer follow-up items (escalate to Sarah)
